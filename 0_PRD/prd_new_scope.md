# PRD New Scope

## 背景

目标是实现 AI 全流程自动化，并能够稳定生成生产级别的代码。前提是假设上游业务语义已经能够相对明确、结构化地传递下来。  
在这个前提下，UX 侧的重点不再只是输出页面设计稿，而是沉淀一套可被 AI 消费、可映射、可约束、可评估的体验资产体系，帮助系统在不同语义场景下匹配到最合适的体验层，并最终产出可上线的代码。

## 目标

- 建立一套面向 AI 自动生成的 UX 资产框架
- 支撑从业务语义到交互模式、页面结构、组件组合、状态处理的稳定映射
- 让生成结果达到生产级要求，而不是停留在 demo 或视觉稿层面
- 为 roadmap 提供明确的 UX 资产准备范围、优先级和阶段划分

## 核心问题

即使上游语义已经较为明确，也仍然存在一个关键问题：  
业务语义并不能直接等于用户体验决策。

在语义到代码之间，UX 侧需要补上一层体验映射机制，解决以下问题：

- 该语义对应什么样的交互模式
- 该场景应该采用什么复杂度和信息密度
- 哪些操作需要引导，哪些可以直接执行
- 哪些任务需要确认、解释、预览、撤销
- 在异常、边界、权限、性能场景下应如何处理

因此，UX 需要准备的不是更多页面，而是一整套结构化资产。

## UX 侧需要准备的资产

### 1. 语义映射资产

目标是把上游需求稳定翻译成体验决策。

需要准备的内容：

- 用户任务模型：用户目标、核心任务、成功标准
- 场景模型：角色、上下文、设备、频率、风险等级
- 任务流与状态机：主流程、分支流程、异常流程、回退路径
- 语义到交互模式映射表：如表单流、配置流、审批流、诊断流、搜索流
- UX decision tree：基于复杂度、风险、熟练度、时长来匹配体验层

价值：

- 帮助 AI 判断“该生成哪种体验”
- 避免功能语义正确，但交互模式错误

### 2. 体验规范资产

目标是限制 AI 的自由发挥范围，确保输出符合统一的体验标准。

需要准备的内容：

- 交互原则：效率优先、可解释优先、风险控制优先、一致性优先等
- 页面级模式规范：列表页、详情页、创建页、编辑页、配置页、空状态页、错误页
- 关键交互规范：搜索、筛选、批量操作、编辑、删除、确认、提交流程
- 反馈规范：loading、success、error、warning、partial success
- 高风险操作规范：二次确认、预览、撤销、权限说明、审批路径

价值：

- 帮助 AI 判断“怎样做才符合 UX 标准”
- 保证不同页面和流程在体验上风格一致

### 3. 设计系统资产

目标是把设计系统从组件库升级为可推理的体验系统。

需要准备的内容：

- 基础 token：颜色、字体、间距、圆角、阴影、动效
- 组件库：Button、Input、Form、Table、Modal、Drawer、Toast、Tabs 等
- 组件使用规则：何时使用 modal、drawer、inline、stepper 等
- Pattern 库：搜索筛选区、表单区、详情区、对比区、结果反馈区、操作栏
- 页面模板：标准 CRUD、审批流、配置流、数据分析流、运营流
- 多端适配规则：桌面、移动、响应式断点下的变体规则

价值：

- 帮助 AI 判断“用什么来构建页面”
- 降低生成结果在结构和视觉层面的离散性

### 4. 生产约束资产

目标是保证生成结果可以上线，而不是只有主路径可用。

需要准备的内容：

- 状态覆盖清单：empty、loading、error、disabled、permission denied、offline
- 边界 case 清单：超长文本、缺失字段、异常数据、重复提交、并发冲突
- Accessibility 规范：键盘可达、焦点管理、语义标签、读屏支持
- i18n 规范：文本膨胀、日期格式、数字格式、RTL 适配
- 性能体验规范：骨架屏、懒加载、渐进式反馈、乐观更新、失败回滚
- 埋点与可观测性规范：关键事件、漏斗节点、错误上报点

价值：

- 帮助 AI 判断“什么才叫生产级”
- 提前显性化传统上依赖人工补齐的隐性要求

### 5. 训练与评估资产

目标是定义什么叫好结果，并让系统持续优化输出。

需要准备的内容：

- 高质量参考样本：需求语义 -> UX 方案 -> 最终代码
- Few-shot 样例库：不同任务类型的标准输入输出
- 反例库：常见错误体验、错误模式选择、缺失状态处理
- UX review checklist：一致性、效率、易懂性、可恢复性、可访问性
- 自动评估 rubric：主流程覆盖、异常流程覆盖、状态完整性、设计一致性

价值：

- 帮助 AI 判断“结果是否合格”
- 为自动化评估、人工 review 和模型优化提供统一基准

### 6. 交付接口资产

目标是让 UX 资产能够被产品、设计、工程和 AI 一起消费。

需要准备的内容：

- 结构化 spec 模板：页面目标、输入、输出、状态、规则、组件依赖
- Figma 规范化文件：命名规则、组件层级、variant、token 映射
- 设计到代码映射：组件名、属性 schema、交互 contract
- Prompt / DSL 模板：把 UX 规则和语义约束喂给 AI 的标准格式
- 版本管理机制：设计系统版本、规范版本、模板版本

价值：

- 帮助 AI 判断“如何稳定读取并执行 UX 资产”
- 降低不同团队之间的语义损耗

## 如何匹配最合适的体验层

要让 AI 在体验层做对选择，不能只依赖 prompt，而要先定义一套匹配机制。

建议基于以下维度进行判断：

- 任务复杂度：低复杂度适合轻量直达，高复杂度适合分步引导
- 风险强度：高风险需要确认、预览、可回退，低风险可以快速执行
- 使用频率：高频任务减少阻力，低频任务增加解释
- 信息不确定性：不确定高时先澄清，不确定低时直接执行
- 用户熟练度：新手偏引导，专家偏效率
- 场景时长：短任务强调快，长任务强调进度感和恢复能力

在此基础上，可以抽象出几类典型体验层：

- 直接操作层：一键执行、内联编辑、即时反馈
- 引导操作层：分步表单、默认推荐、过程解释
- 审核确认层：预览、差异对比、二次确认、审批流
- 策略配置层：规则设置、自动化编排、批量处理
- 诊断恢复层：错误定位、原因解释、修复建议、重试路径

UX 侧需要产出的关键不是某一种页面，而是“在什么条件下选哪一种体验层”的决策规则。

## Roadmap 建议分阶段

### Phase 1: 基础标准化

目标：先建立可复用、可约束的底座。

重点资产：

- 设计 token
- 基础组件库
- 页面模板
- 核心交互规范
- 状态清单

### Phase 2: 结构化映射

目标：建立从语义到体验的稳定转换机制。

重点资产：

- 用户任务模型
- 场景模型
- 任务流与状态机
- 语义到交互模式映射表
- UX decision tree

### Phase 3: 生产化完善

目标：让生成结果具备真实上线能力。

重点资产：

- Accessibility 规范
- i18n 规范
- 性能体验规范
- 异常和边界 case 规范
- 埋点与可观测性规范

### Phase 4: 智能化优化

目标：让系统具备持续生成、持续评估、持续优化的能力。

重点资产：

- 高质量样本库
- Few-shot 示例
- 反例库
- 自动评估 rubric
- Prompt / DSL 模板

## 建议的资产清单视图

为了方便 roadmap 管理，建议后续将所有 UX 资产整理成统一表格，字段可包括：

- Asset 名称
- 资产类型
- 目标
- 具体产出物
- 使用方
- 优先级
- 所属阶段
- Owner
- 当前状态

## 一句话总结

为了实现 AI 全流程自动化并生成生产级代码，UX 侧需要准备的不是单点设计稿，而是四层核心能力：

- 决策资产：语义如何映射到体验层
- 规范资产：体验应该如何被约束
- 系统资产：界面如何被稳定构建
- 评估资产：结果如何被判断为生产级

这套资产体系将决定 AI 最终生成的不是“看起来像产品的页面”，而是真正可交付、可上线、可维护的产品体验和代码。
