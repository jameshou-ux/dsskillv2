四 · Token 语义四层结构

设计资产的分层治理

从原子值到动态响应，四层语义逐级抽象

01
Primitive Layer · 基础层
定义最基础的原子值（颜色、间距、圆角等 scale），不包含业务语义，是系统的视觉比例尺。
primitive.color.blue.500
02
Semantic Layer · 语义层
核心 AI 理解层，遵循 category.role.state 命名。将基础值映射到设计意图，支持跨场景推理。
semantic.text.primary.active
03
Pattern Layer · 模式层
抽象通用的 UI 模式（如 Surface、Layout），定义更高维度的交互样式规则。
pattern.surface.card
04
Component Layer · 组件层
组件级的实例化决策。组合语义与模式，用于 Runtime 与 AI 系统重建具体组件。
component.button.primary.bg
四 · AI Token 消费流

AI-Native 工作流与规则限制

AI 从接受 Prompt 到重组组件级的强一致性约束

🔄 AI链路 (Generation Flow)
1. User Prompt
▼
2. AI Reasoning (意图与语义解析)
▼
3. Component Resolution (解析组件层)
▼
4. Token Lookup (检索 Semantic/Pattern)
▼
5. Generate UI (按需调用代码或设计模型)
🧩 AI 重组组件 (Resolution)
AI 通过 Token 层级重建组件。读取 component.button.primary 时，AI 会自动拆解：
  • ✨ Background: 映射至对应的 Semantic 色彩
  • ✨ Typography: 匹配标准文本样式
  • ✨ Spacing/Layout: 精确对应至 Primitive 比例尺
🛡️ AI 设计强一致性约束
AI 生成 UI 必须严格匹配设计规范，禁止创造未知视觉偏差：
  • ❌ 禁止硬编码颜色: 严禁直接书写 HEX (如 #007FFF)
  • ❌ 禁止魔法数字: Spacing 必须来自 Scale (如 7px → spacing.8)
  • ✅ 绝对映射: 原则上只应生成结构中预定义的引用和数值
五 · 原因 & 决策背景

关键技术决策

为什么选择 SSOT + Adapter 路线

🔵 Figma MCP 与 API 能力限制
Figma MCP 的 get variable 无法获取全局 Variable Token 定义，仅能获取页面内 Node 应用的变量。REST API 写入变量 (file_variables:write) 为 Enterprise 专属权限，成本效益较低。
🔵 Token Schema 差异与兼容性
Figma 原生 Schema 不支持 Gradient 等复杂属性；TokenStudio 完整 Schema 需 Pro 计划。这些格式与四层语义结构存在差异 → 决定采用 SSOT + Adapter Layer 架构，在应用层统一转换。
