Skill README

R2D2@DS

这是一个面向 imToken 内部设计系统的治理型技能。它不替代底层技能，而是负责先判断任务落在哪个场景，再决定如何串接子技能，完成 DS 治理、DS 到设计、设计到 Demo、Prompt 到 Demo，以及 DS 到代码的工作。

目的：DS 治理 + DS 驱动产出 执行底座：figma-use 核心：audit-design-system + figma-generate-library

技能目的

它解决什么问题

用它来治理一个 messy 或正在演进中的设计系统，把设计真正接回 DS，基于设计或 Prompt 生成 Demo，并在条件成熟时推进到代码 handoff。

Governance Application Design To Demo Prompt To Demo DS To Code

它不是什么

它不是自由发挥的设计生成器，也不是直接输出生产代码的单一工具。它更像一个协调器，负责选择技能链路和执行顺序。

不是泛化设计生成器 不是直接 codegen 工具

涉及子技能

必需

  • figma-use：Figma 读写执行底座
  • audit-design-system：发现具体的 DS drift
  • figma-generate-library：判断当前所处阶段与工作顺序

按需调用

  • sync-figma-token：token source 与 Figma 对账
  • apply-design-system：section/page 级接回 DS
  • rad-spacing：层级与间距治理
  • clarify / normalize：命名、文案、规范收敛

下游阶段

  • figma-generate-design / edit-figma-design：基于 DS 的设计生成
  • figma-implement-design：Figma 节点到代码翻译
  • figma-code-connect-components：组件到代码映射
  • cc-figma-*：只用于成熟 family 的 regenerate

场景与流程

1. DS Governance

目标：让 DS 变得完整、可读、稳定、可治理。

阶段

figma-generate-library 先判断当前问题属于 foundations、component hardening、page canonization 还是下游任务。

审查

audit-design-system 找出 drift、裸值、弱命名、未绑定 token/style 等问题。

执行

figma-use 做小步修复。只有涉及 source 对齐时才调用 sync-figma-token。

例子

你有一个 DS 文件，里面 variable 很多但 collection 结构混乱，组件还在使用 local fill 和裸 spacing。这个技能会先把任务归类到 governance，再审查具体 drift，最后做小步修复，而不是直接进入原型生成或代码交付。

2. DS Application

目标：把 DS 应用到具体产物上。这里不是单一模式，而是一层应用层分流。

分流

进入应用层后，必须继续归类到四个子场景之一：DS To Design、Design To Demo、Prompt To Demo、DS To Code。

规则

应用层的关键不是“看起来像 DS”，而是“实际使用 DS”。优先复用真实 DS 组件；没有现成业务组件时，先做最小本地可复用组件，再用实例组装。

验证

应用层结果必须做结构、截图和关键节点数据三层验证，不能只看视觉截图，也不能只看节点存在。

例子

一个任务说“基于现有 DS 做 send token flow”。这还不够，技能需要继续判断到底是要写回 Figma、做 Demo、还是直接进入代码 handoff，然后再选对链路。

3. DS To Design

目标：基于 DS 生成、修复或扩展 Figma 设计，让设计本身进入真实 DS-backed 状态。

检查

audit-design-system 先确认页面或页面组里哪些地方没有真正使用系统组件、token、style，或者结构漂移。

重连

apply-design-system 负责 section/page 级接回 canonical primitives。figma-use 负责局部 edit、绑定、组件替换和验证。

收尾

rad-spacing、clarify、normalize 用于 spacing、copy、命名和规范收敛。目标不是视觉接近，而是让设计在 Figma 里真正使用 DS。

例子

一个产品页面是随手搭建的，按钮、卡片、标签都有，但没有绑定 DS variables 或 styles。这个技能会先审查，再把 color、string、dimension、font styles、组件实例和结构一起接回 DS。

4. Design To Demo

目标：以现有设计作为主输入，生成非 production 的 Demo。它需要真实使用 DS，而不是只做视觉模仿。

输入

设计稿或 DS-backed Figma surface 是主要来源。输出是 prototype、sample implementation 或交互 Demo，而不是最终生产实现。

DS 使用

Demo 必须优先复用真实 DS 组件。没有现成组件时，先做最小本地可复用组件，再用实例组装，而不是重复画 primitives 或只做 token/style 表层对齐。

边界

它和 DS To Code 的区别在于：这里追求可用 Demo 和正确 look & feel，不追求 production-ready 结构与 handoff 质量。

例子

你给一张 send token flow 设计稿，要求生成一个 mobile Demo。这个技能会把它视为 design-to-demo：设计是主要输入，结果必须复用 DS，而不是只在视觉上“像”。

5. Prompt To Demo

目标：以 prompt、PRD 或产品意图为主输入，直接生成 Demo。默认输出是 Demo，不是 Figma。

输入

主要输入是 prompt、PRD、设计 brief 或产品意图。系统需要先把意图翻译成设计要求，再去选 DS surfaces、组件和 token。

默认输出

Prompt To Demo 默认是 demo-oriented。若用户最终要的是 Figma 设计稿，应改路由到 DS To Design，不能在这里默认偷换输出面。

DS 使用

规则与 design-to-demo 相同：优先真实 DS 组件，其次最小本地可复用组件；token/style 是基础层，不足以单独证明“用了 DS”。

例子

你给出一个“账户安全页”的 prompt，再给一套稳定的 DS，包括 navigation、cards、buttons 和 typography tokens。这个技能会把它视为 prompt-to-demo：先基于 DS 生成 Demo，再做 post-generation audit 和验证。

6. DS To Code

目标：从稳定的 canonical Figma surface 或稳定的 DS family 进入代码准备，而不是从历史治理板直接 codegen。

门槛

目标页面或组件 family 必须已经 canonical、命名稳定、token 绑定清楚。历史 cleanup board 不能直接作为 handoff surface。

翻译

figma-implement-design 用于 node-to-code fidelity，figma-code-connect-components 只在 component mapping 成熟时才使用。

回退

如果 family 还不稳定，就回退到 governance 或其他 application 子场景，不强行进入 code workflow。

例子

你想从一个导航 family 生成代码。如果这个 family 已有稳定 variants、token bindings 和 canonical page ownership，就可以进入 DS-to-code。如果这些条件不满足，技能会要求先继续 harden Figma 侧。

规则

必需输入

  • 当前 DS 的 canonical source token JSON
  • 相关的 Figma 文件或页面
  • 当结构和对账相关时，需要 adapter 或 governance schema
  • 当涉及 design generation 时，需要 intent source

停止条件

  • 缺少必需 skill
  • 缺少必需输入
  • 页面角色不清
  • DS 应用信心不足
  • 当前 surface 还不适合进入 code handoff

如果缺少必需 skill 或输入，这个技能必须明确告诉用户缺什么、为什么缺、以及需要先补什么，而不是继续猜。
