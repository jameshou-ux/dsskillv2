Figma to Code 视角

AI to Code 视角

从 agent 实现角度看，当前状态比之前更可用，因为 component naming、variant schema、text style 和 local variable binding 已明显收敛。但如果直接大规模进入 AI to Code，仍然会遇到 canonical page 尚未完全定型的问题，所以更适合先围绕少量页面和成熟 family 做试点，而不是全量铺开。

better for retrieval less naming drift not full-scale ready
结论上，当前 DS 已经从“仅适合治理”推进到“可以开始局部 handoff 和局部 AI implementation 试点”，但还没有到“所有目标页都能稳定作为 code-side source of truth”的阶段。

新 Skill Workflow

执行底座

`figma-use` 是实际 Figma 操作底座，承担页面读写、变量绑定、节点修改和结构迁移。

hard dependency

治理框架

`figma-generate-library` 负责 phase 顺序和 DS 结构原则，决定先 foundations、后 components、再 QA 的治理逻辑。

phase logic workflow orchestration

专项治理

`audit-design-system`、`sync-figma-token`、`rad-spacing`、`apply-design-system` 负责 drift 诊断、token 对齐、间距规范化和页面级重连。

optional but high value

当前主组合

当前主线实际使用的是 `figma-use` + `figma-generate-library` + `audit-design-system` + `sync-figma-token` + `rad-spacing`。这是面向现有 messy DS 文件的治理流，不是 contract-first 生成流。

governance-first existing-file flow

暂不是主路径

`cc-figma-tokens`、`cc-figma-component` 更适合 contract 成熟后的 regenerate；`figma-code-connect-components`、`figma-generate-design`、`edit-figma-design` 更适合治理之后的下游工作流。

downstream later-stage

Workflow 展示

01 · Inspect

`figma-use`

盘点页面、变量、样式和组件结构，建立当前 Figma 文件的真实状态。

02 · Audit

`audit-design-system`

识别 drift、local wrapper、未绑定 token/style 以及 variant schema 问题。

03 · Token Parity

`sync-figma-token`

在 code token 和 Figma variable 之间做 dry-run parity 检查，补缺口、修 alias、修 scope 与 syntax。

04 · Targeted Write

`figma-generate-library` + `figma-use`

按 foundations → file structure → components 的顺序做增量治理，一次只改一个 page、一个 family 或一组 tokens。

05 · Normalize

`apply-design-system` / `rad-spacing`

在页面或 section 级做系统化重连，并对 gap、padding、层级节奏做规范化。

06 · Validate

Re-check and canonize

验证当前页面结构是否已经适合从治理页过渡到 canonical component pages。

下一步计划

页面层

  1. 按页硬化 `原子组件/Button`、`原子组件/Navigation`、`原子组件/Modal`。
  2. 明确哪些页是 canonical edit surfaces，哪些页只保留为治理历史。
  3. 继续让 `Semantic` 成为默认 binding layer，减少 page-local 例外。

Canoical 化之后

  1. 把 canonical component pages 作为正式 handoff surface。
  2. 再接 `Figma to Code` 与 `AI to Code`。
  3. 在成熟 family 上再考虑 `Code Connect` 与 `cc-*` regenerate 流。

结论

当前文件已经从“主要依赖治理板推进”进入“治理板 + canonical 页面候选并存”的状态。接下来最重要的不是继续加更多治理动作，而是把 `Button`、`Navigation`、`Modal` 这类目标页正式扶正，形成明确的 canonical edit surface，再承接后续的 handoff 和 code-side workflow。
