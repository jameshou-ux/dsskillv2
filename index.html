三 · 架构挑战与原因

为何采用当前架构

应对平台限制与标准隔离带来的挑战

🔵 Figma MCP 与 API 能力限制
  • Figma MCP 的 get variable 无法获取全局 Variable Token 定义，仅能获取页面内 Node 应用的变量。
  • REST API 写入变量 (file_variables:write) 为 Enterprise 专属权限，且 升级 Enterprise 也无法解决所有自动化问题：
    • API 能力缺失: 官方 REST API 至今不支持写入/更新 Styles（渐变色、阴影特效等），仍然会有同步断层，无法实现真正的服务端全自动化闭环。
    • 投入产出比低: 仅为了获取变量写入 API 而承担倍增的 Enterprise 席位成本（ROI 极低）。
  • 精准度问题: MCP 的变量读取局限意味着，普通的 Agent Browser Audit 不能精准读取整套全局设计规范，导致大模型直接生成的 Audit 报告容易失真。
↳ 核心定论：当前 Org/Pro Plan + Plugin API 自动化脚本（结合 CI/CD） 是打通 Variables + Styles 最具性价比且最高效的可行途径。
🔵 Token Schema 差异与兼容性
Figma 原生 Schema 较为保守（不支持 Gradient 等）且与其它插件 Schema（TokenStudio）标准存在差异。四层语义必须统一范式。
↳ 决定采用 SSOT + Adapter Layer 架构，在应用层统一转换与分流。

四 · Token 语义四层结构

设计资产的分层治理

从原子值到动态响应，四层语义逐级抽象

01
Primitive Layer · 基础层
定义最基础的原子值（颜色、间距、圆角等 scale），不包含业务语义，是系统的视觉比例尺。
primitive.color.blue.500
02
Semantic Layer · 语义层
核心 AI 理解层，遵循 category.role.state 命名。将基础值映射到设计意图，支持跨场景推理。
semantic.text.primary.active
03
Pattern Layer · 模式层
抽象通用的 UI 模式（如 Surface、Layout），定义更高维度的交互样式规则。
pattern.surface.card
04
Component Layer · 组件层
组件级的实例化决策。组合语义与模式，用于 Runtime 与 AI 系统重建具体组件。
component.button.primary.bg

四 · AI Token 消费流

AI-Native 链路与一致性约束

AI 从接受 Prompt 到重组组件级的强一致性规范

🔄 生图链路 (Generation Flow)
1. User Prompt
▼
2. AI Reasoning (意图与语义解析)
▼
3. Component Resolution (解析组件层)
▼
4. Token Lookup (检索 Semantic/Pattern)
▼
5. Generate UI (基于模型预设产出)
🧩 AI 重组组件 (Resolution)
AI 通过 Token 层级重建组件。读取 component.button.primary 时，会自动拆解并映射：
  • ✨ 色彩/阴影: 对应 Semantic Token
  • ✨ 字符/段间距: 映射 Primitive 比例尺
🛡️ 强一致性设计约束
AI 生成 UI 必须严格匹配设计规范，禁止创造未知视觉偏差：
  • ❌ 禁止颜色硬编码: 严禁直接写死色值 (#007FFF)
  • ❌ 禁止魔法数字: 必须通过 Scale 指定 (7px → spacing.8)
  • ✅ 绝对映射: 要求对预定义资产的绝对映射应用
