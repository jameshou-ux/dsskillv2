面向 AI 全流程自动化的
UX 资产体系

目标不是生成"看起来像产品的页面"，而是建立一套可被 AI 消费、可映射、可约束、可评估的体验资产体系，支撑从业务语义到可上线代码的全链路稳定生成。

一句话总结
决策资产
语义如何映射到体验层
规范资产
体验应该如何被约束
系统资产
界面如何被稳定构建
评估资产
结果如何判断为生产级

核心问题

即使上游业务语义已经结构化传递，业务语义并不等于用户体验决策。语义到代码之间缺少一层体验映射机制：

该语义对应什么样的交互模式
该场景应采用什么复杂度和信息密度
哪些操作需要引导，哪些可以直接执行
哪些任务需要确认、预览、撤销
在异常、边界、权限、性能场景下应如何处理

→ UX 需要准备的不是更多页面，而是一整套结构化资产。

01
语义映射资产
帮助 AI 判断"该生成哪种体验"

把上游需求稳定翻译成体验决策，避免功能语义正确但交互模式错误。

用户任务模型：目标、核心任务、成功标准
场景模型：角色、上下文、设备、频率、风险等级
任务流与状态机：主流程 / 分支流程 / 异常流程 / 回退路径
语义到交互模式映射表（表单流、配置流、审批流、诊断流…）
UX decision tree：基于复杂度、风险、熟练度、时长匹配体验层

02
体验规范资产
帮助 AI 判断"怎样做才符合 UX 标准"

凌驾于组件之上的行为规则层 — 回答"该怎么做"，与具体用什么组件无关，在任何组件库下都成立。

交互原则：效率 / 可解释 / 风险控制 / 一致性
页面级模式规范：列表页、详情页、创建页、编辑页、配置页、空状态页、错误页
关键交互规范：搜索、筛选、批量操作、编辑、删除、确认、提交流程
反馈规范：loading / success / error / warning / partial success
高风险操作规范：二次确认、预览、撤销、权限说明、审批路径
具体产出样例
interaction-principles.yaml · 交互原则
principles:
  - id: risk-control
    rule: "高风险操作必须二次确认，不可直接执行"
    applies_to: [delete, transfer, batch-submit]
  - id: efficiency
    rule: "高频任务路径不超过 2 步，禁止冗余确认"
    applies_to: [view, filter, copy]
  - id: explainability
    rule: "AI 生成结果必须附带来源说明，用户可查看依据"
    applies_to: [ai-generated-content]
page-pattern.yaml · 页面级模式规范（列表页）
page: list
required_zones:
  - search-filter-bar
  - data-table
  - pagination
  - empty-state
  - batch-action-bar   # 仅勾选时出现
optional_zones:
  - side-panel         # 详情预览
rules:
  - 空状态必须包含引导操作，不能只显示"暂无数据"
  - 批量操作超过 20 条需显示进度，不能 block UI
feedback-spec.yaml · 反馈规范
feedback:
  loading:
    duration: "< 300ms: 无反馈  |  300ms-3s: spinner  |  >3s: 骨架屏"
  success:
    default: Toast (3s auto-dismiss)
    critical: 全屏成功页（如转账完成）
  error:
    inline: 字段级错误，红色提示紧跟输入框
    page: 操作失败时保留已填内容，提供重试入口
  partial_success:
    rule: "批量操作部分失败时，逐条列出失败原因"
high-risk-spec.yaml · 高风险操作规范
high_risk_ops:
  delete:
    require: [二次确认弹窗, 明确说明影响范围]
    allow_undo: true
    undo_window: 5s
  transfer:
    require: [金额预览, 地址核对, 二次确认]
    require_preview: true
  batch_submit:
    require: [差异对比, 影响数量说明]
    max_auto_execute: 10   # 超出必须分批确认
03
设计系统资产
帮助 AI 判断"用什么来构建页面"

回答"用什么来实现" — 提供具体构建材料。与体验规范（02）协同：02 定规则，03 给工具。

基础 token：颜色、字体、间距、圆角、阴影、动效
组件库：Button、Input、Form、Table、Modal、Drawer、Toast、Tabs…
组件使用规则：何时用 modal / drawer / inline / stepper
Pattern 库：搜索筛选区、表单区、详情区、对比区、结果反馈区、操作栏
页面模板：标准 CRUD、审批流、配置流、数据分析流、运营流
多端适配规则：桌面 / 移动 / 响应式断点变体

component-rules.yaml · 组件使用规则
component_selection:
  - trigger: "需要打断用户当前流程并要求确认"
    use: AlertDialog
    avoid: Toast

  - trigger: "展示详情，用户仍需保留上下文"
    use: Drawer (side panel)
    avoid: Modal

  - trigger: "多步骤任务，步骤间有依赖"
    use: Stepper
    avoid: Tabs

  - trigger: "行内快速编辑，不需要跳转"
    use: Inline Edit
    avoid: 跳转到编辑页
pattern-search-filter.yaml · Pattern 库：搜索筛选区 
pattern: search-filter-bar
slots:
  - name: keyword-input
    component: Input
    props: { placeholder: "搜索...", clearable: true }
  - name: filter-group
    component: Select / DateRangePicker / CheckboxGroup
    layout: inline-wrap
  - name: reset-btn
    component: Button
    variant: ghost
    show_when: any_filter_active
behavior:
  - 筛选变更即时触发，无需点击"搜索"按钮
  - 筛选条件以 Tag 形式展示，可单独删除

04
生产约束资产
帮助 AI 判断"什么才叫生产级"

保证生成结果可以上线，不只是主路径可用，显性化传统上依赖人工补齐的隐性要求。

状态覆盖清单：empty / loading / error / disabled / permission denied / offline
边界 case：超长文本、缺失字段、异常数据、重复提交、并发冲突
Accessibility：键盘可达、焦点管理、语义标签、读屏支持
i18n：文本膨胀、日期 / 数字格式、RTL 适配
性能体验：骨架屏、懒加载、渐进式反馈、乐观更新、失败回滚
埋点与可观测性：关键事件、漏斗节点、错误上报点
具体产出样例
state-checklist.yaml · 状态覆盖清单
component: TransferForm
states:
  empty:
    trigger: 页面初始加载
    display: 输入框 placeholder，主按钮 disabled
  loading:
    trigger: 提交中
    display: 按钮 loading spinner，表单字段 disabled
  error:
    trigger: 网络失败 / 余额不足
    display: 行内错误提示，保留已填内容，显示重试按钮
  disabled:
    trigger: 用户无转账权限
    display: 表单置灰，顶部 Banner 说明原因
  permission_denied:
    trigger: 钱包未连接
    display: 空状态页，引导连接钱包
  offline:
    trigger: 网络断开
    display: 顶部 Toast 警告，禁止提交
edge-cases.yaml · 边界 Case 清单
edge_cases:
  - scenario: 超长地址输入
    expect: 截断显示 + tooltip 展示完整，不撑破布局
  - scenario: 金额输入超过余额
    expect: 实时校验，红色提示，按钮保持 disabled
  - scenario: 重复点击提交
    expect: 第一次点击后立即 disabled，防止重复提交
  - scenario: 并发冲突（同一资产被他人操作）
    expect: 提交时返回冲突提示，说明原因，提供刷新入口
  - scenario: 缺失必填字段提交
    expect: 逐字段标红，滚动至第一个错误位置
a11y-spec.yaml · Accessibility 规范
rules:
  - id: keyboard-nav
    rule: 所有交互元素可通过 Tab 键到达，顺序符合视觉阅读顺序
  - id: focus-management
    rule: Modal 打开时焦点移入，关闭后焦点回到触发元素
  - id: aria-labels
    rule: 图标按钮必须有 aria-label，表单字段必须关联 label
  - id: color-contrast
    rule: 正文对比度 ≥ 4.5:1，大号标题 ≥ 3:1（WCAG AA）
  - id: screen-reader
    rule: 动态更新内容使用 aria-live，错误提示使用 role="alert"
performance-spec.yaml · 性能体验规范
loading_strategy:
  skeleton_screen:
    use_when: 内容结构已知，加载时间预计 > 300ms
    avoid_when: 数据量极小或加载极快（< 100ms）
  optimistic_update:
    use_when: 操作成功率 > 95%，且可回滚
    example: 点赞、收藏、简单状态切换
  progressive_feedback:
    rule: 长任务（> 3s）必须有进度指示，不能只用 spinner
  lazy_load:
    rule: 列表图片、非首屏内容统一懒加载
  rollback:
    rule: 乐观更新失败时，自动还原 UI 并提示用户
05
训练与评估资产
帮助 AI 判断"结果是否合格"

定义什么叫好结果，为自动评估、人工 review 和模型优化提供统一基准。

高质量参考样本：需求语义 → UX 方案 → 最终代码
Few-shot 样例库：不同任务类型的标准输入输出
反例库：常见错误体验、错误模式选择、缺失状态处理
UX review checklist：一致性、效率、易懂性、可恢复性、可访问性
自动评估 rubric：主 / 异常流程覆盖、状态完整性、设计一致性
具体产出样例
few-shot-sample.yaml · Few-shot 样例
- input:
    semantic: "用户发起一笔 ETH 转账，金额较大，目标地址首次使用"
    risk: high
    frequency: low
    expertise: intermediate
  output:
    layer: 审核确认层
    flow:
      - 输入金额和地址
      - 地址风险检测（首次使用提示）
      - 金额 + 手续费预览页
      - 二次确认（密码或生物识别）
      - 提交 → loading → 成功/失败
    components: [Input, AddressInput, PreviewCard, ConfirmDialog, Toast]
    must_have: [地址校验, 金额预览, 二次确认, 失败重试]
anti-patterns.yaml · 反例库
anti_patterns:
  - id: empty-state-no-action
    wrong: 列表为空时只显示"暂无数据"文字
    correct: 提供引导操作（如"创建第一条记录"按钮）
    reason: 空状态是引导用户的机会，不是终点

  - id: error-message-generic
    wrong: 所有错误统一显示"操作失败，请重试"
    correct: 区分错误类型，说明原因并给出具体操作建议
    reason: 用户需要知道"为什么失败"才能正确处理

  - id: modal-overuse
    wrong: 详情展示用 Modal，打断用户上下文
    correct: 用 Drawer 或 Side Panel，保留列表可见
    reason: 用户在查看详情时通常需要对比其他条目

  - id: missing-loading-state
    wrong: 点击提交后无反馈，用户不知道是否成功
    correct: 立即切换按钮为 loading 状态
    reason: 无反馈会导致重复提交
ux-review-checklist.yaml · UX Review Checklist
checklist:
  consistency:
    - 同类操作使用相同组件和交互模式
    - 术语和标签在全产品中一致
  efficiency:
    - 高频任务路径 ≤ 2 步
    - 无冗余确认步骤
  clarity:
    - 每个页面有明确的主操作
    - 错误信息说明原因和解决方式
  recoverability:
    - 高风险操作可撤销或有回退路径
    - 表单提交失败后保留已填内容
  accessibility:
    - 键盘可完整操作
    - 色彩对比度达标
eval-rubric.yaml · 自动评估 Rubric
rubric:
  - dimension: 主流程覆盖
    score_4: 主流程完整，无缺失步骤
    score_2: 主流程存在跳步或逻辑断点
    score_0: 主流程不可用

  - dimension: 异常流程覆盖
    score_4: error / empty / loading 全部处理
    score_2: 仅处理 error，缺少 empty 或 loading
    score_0: 无任何异常处理

  - dimension: 设计一致性
    score_4: 组件选用符合规范，无自定义偏差
    score_2: 存在 1-2 处组件误用
    score_0: 大量偏离设计系统
06
交付接口资产
帮助 AI 判断"如何稳定读取并执行 UX 资产"

让 UX 资产能够被产品、设计、工程和 AI 一起消费，降低不同团队之间的语义损耗。

结构化 spec 模板：页面目标、输入、输出、状态、规则、组件依赖
Figma 规范化文件：命名规则、组件层级、variant、token 映射
设计到代码映射：组件名、属性 schema、交互 contract
Prompt / DSL 模板：将 UX 规则和语义约束喂给 AI 的标准格式
版本管理机制：设计系统版本、规范版本、模板版本

体验层匹配机制

基于以下维度判断匹配哪一种体验层：

任务复杂度
低 → 轻量直达 / 高 → 分步引导
风险强度
高 → 确认 / 预览 / 可回退
使用频率
高频 → 减少阻力 / 低频 → 增加解释
信息不确定性
高 → 先澄清 / 低 → 直接执行
用户熟练度
新手 → 引导 / 专家 → 效率
场景时长
短 → 快 / 长 → 进度感 / 恢复能力
五类典型体验层
直接操作层 — 一键执行、内联编辑、即时反馈
引导操作层 — 分步表单、默认推荐、过程解释
审核确认层 — 预览、差异对比、二次确认、审批流
策略配置层 — 规则设置、自动化编排、批量处理
诊断恢复层 — 错误定位、原因解释、修复建议、重试路径
